配信と利用環境ガイド

「私（保守者）の利用環境」と「ユーザーの利用環境」がどう違うのか、ユーザーがどうセットアップするのか、データベース（D1）はどう立つのかを、図とともに要点解説する文書です。出典は本リポジトリの設定ファイル・ADR と Cloudflare 公式ドキュメント（末尾に一覧）。Cloudflare の挙動に関する記述は公式ドキュメント・GitHub で確認したものですが、配布の通し挙動の最終確認は実デプロイで行う前提です。

1. 全体像：4つのフロー

このプロジェクトには性質の異なる4つのフローがあります。①開発のあと道が左右に分かれ、③私の本番はローカルからCLIで直接、②配信→④ユーザーの本番は公開GitHub経由のノンCLIで構築されます。いったん動き出せば、運用と利用の方法は両者で同じです。

  • 違いは「どう動かし始めるか・どう更新するか」だけ。 動き出した後の運用（管理UI）と利用（Discord）は両者で同一です。
  • ②配信は「届ける仕組み」で、利用そのものではありません。BOOTH（入口）＋公開GitHub（Deployボタンの動力源）の二層構成（ADR 0011）。
  • インスタンスは人ごとに別物。 それぞれ別のCloudflareアカウント・別のWorker・別のD1で、データは完全に独立（テナント分離ではなく各自が自己ホスト）。

2. 私の利用環境 vs ユーザーの利用環境

核心は「ソースコードの居場所」と「デプロイ経路」が左右で対になっている点です。私のソースの正本はローカル（PC）にあり、それを公開GitHubへ push して配布元にします。ユーザーのソースは公開GitHubを fork した「ユーザー自身のGitHub上のコピー」で、ローカルには持ちません。

軸私の利用環境ユーザーの利用環境
ソースの正本ローカル（私のPC）GitHubのfork（ユーザーのGitHub上）
公開GitHubとの関係私が所有（オリジナル）・push するオリジナルを fork して使う
デプロイ手段npm run deploy（CLI・手動）Workers Builds（自動・ノンCLI）
Cloudflare自分で wrangler d1 create 等で自前構築Deployボタンで自動生成（利用者ごとに別名・名前は環境依存）
シークレット投入wrangler secret put（CLI）Deployボタンの入力欄に貼り付け
マイグレーション私がCLIで本番D1へ（--remote・要許可）deployスクリプトが各自のD1へ自動適用
更新のきっかけ私がもう一度 npm run deployユーザーが fork で「Sync fork」
Discord本番Choiemu（私のサーバー）ユーザー各自のサーバー

3. ユーザーのセットアップとD1の自動構築

ユーザーが必要とするのは無料の3アカウントだけ。データベース（D1）はユーザーが手で作る場面が一切なく、Deployボタンが自動生成し、スキーマもマイグレーションで自動適用されます。

必要なアカウント用途費用
DiscordBotアプリ＋導入先サーバーの管理権限無料
Cloudflareホスティング（Worker）＋データベース（D1）無料枠
GitHubDeployボタンが作る fork の置き場＋更新（Sync fork）無料

D1がゼロ操作で立つ仕組み（Q への答え）

Deployボタンが wrangler 設定のD1バインディング宣言を読み、ユーザーのアカウントにD1を自動生成します（後述リスク①のとおり、自動生成には database_id を空にした配布用設定が前提）。デプロイ時に deploy スクリプトが wrangler d1 migrations apply DB --remote を先に走らせ、migrations/ の全連番（v7.0.1 時点で 0001〜0009）を順に適用してスキーマを自動構築します。マイグレーションを「DB名」でなくバインディング名 DB で指定しているのは、ユーザーごとにD1名が変わっても同じスクリプトで動かすためです。なお「合言葉」とは Deployボタンの入力欄で渡す4つのシークレット（DISCORD_PUBLIC_KEY / DISCORD_APPLICATION_ID / DISCORD_BOT_TOKEN / ADMIN_TOKEN。出典 .dev.vars.example ／ package.json の cloudflare.bindings）のことです。

「ローカル→Cloudflare」でGitHubを省けないのか？

省けますが、GitHub登録（Webフォーム1回）の代わりにCLI環境一式（Node＋wrangler＋ターミナル操作）を背負わせることになり、非エンジニアには逆に難しくなります。Cloudflareで「Worker＋D1＋マイグレーション」をノンCLIで立てる経路は実質Deployボタンだけで、それは公開gitリポジトリを参照するため、GitHub（またはGitLab）は原理的に外せません。さらにGitHubは「更新がSync fork 1回」も成立させているので、継続的にも効きます。結論：3サービス（GitHub/Cloudflare/Discord）が最小かつ最簡。

4. リスク・要対応

公開配布を本当に成立させる前に、以下を確認・対応する必要があります。重大度は確度×影響で付与（最優先＝放置すると配布が成立しない／中＝品質・信頼性の問題／対応済み＝本変更で修正）。①②は「Cloudflare 仕様上ほぼ確実に詰まる／設定が要る」未対応項目、③（記述不整合）は対応済みです。

最優先 ① database_id のベタ書き

事実wrangler.toml に私のアカウントの実 database_id（f6ff753c-…）がベタ書きされている（wrangler.toml:11）。
影響Cloudflareの自動プロビジョニングは「IDが書かれているバインディング＝既存リソース」とみなして新規作成をスキップする（Cloudflare公式・末尾出典で確認）。そのIDはユーザーのアカウントに存在しないため、デプロイが失敗する蓋然性が高い（最終確認は実デプロイ）。
対応配布用は database_id を空/省略にして自動プロビジョニングを発火させる。さらに、自動生成IDの書き戻しが .toml では行われない（issue #13632）ため、wrangler.jsonc への移行が無難。マイグレーション側は PR #14275（wrangler 4.102.0 で初収録）でバインディング名からUUIDをAPI解決するようになり、ID空でも wrangler d1 migrations apply DB --remote が通る。本プロジェクトは解決版 4.103.0 で条件を満たす。ただし宣言レンジが ^4.101.0 のままだと lockfile 再生成時に #14275 未収録の 4.101.x へ戻りうるため、floor を ^4.102.0 へ引き上げるのが望ましい（要対応）。※ package.json 変更時は @emnapi ピン留め（Deployビルド対策）を崩さないよう、lockfile を npm10 で再生成して npm ci 整合を保つこと。
留意同じ wrangler.toml をローカルでも使うため、database_id を空にすると私のローカル npm run deploy も自動プロビジョニングで別のD1を作り、本番Choiemuと違うDBに繋がってしまう（＝本番データから切り離される）。回避策は2つ — (a) §5の3層案で本番もWorkers Builds経路に寄せ、ローカルからのリモートデプロイ自体を廃止する（推奨。これなら矛盾が消える）／(b) ローカルだけ database_id を含む別設定（例 wrangler.local.toml）で上書きデプロイする。
出典との関係ADR 0011／dev-and-release-flow.md が「未検証・初回実デプロイで確認」としているのは「Deployボタン配布時のD1別名生成＋マイグレーション自動適用の通し動作」であって、database_id の空化や jsonc 化には触れていません。本リスク①は、その通し検証を成立させる前提として本ガイドが新たに指摘する論点です。

最優先 ② Workers Builds のデプロイコマンド設定

ADR 0011・dev-and-release-flow.md は「Workers Builds は本番ブランチへの push ごとに deploy コマンドを実行する」前提に立っていますが、その既定コマンドは npx wrangler deploy（マイグレーションを実行しない）です（Cloudflare公式・末尾出典）。本プロジェクトはマイグレーション込みの npm run deploy（package.json:9）を走らせる必要があるため、Workers Builds のビルド設定でデプロイコマンドを npm run deploy に上書きしなければなりません。未設定だと Sync fork してもスキーマ更新が走らず不整合になります（確定事実なので①と同格・最優先）。

対応済み ③ setup.html の「用意するもの」にGitHubが無い（修正済み）

「用意するもの」が Discord・Cloudflare の2つだけで GitHub を欠いていた（同ファイルの更新手順では fork の Sync fork を要求しており自己矛盾）。本変更で setup.html・README.md・docs/dev/dev-and-release-flow.md・.claude/rules/dev-and-release.md を「Discord・Cloudflare・GitHub の3つ」に統一修正済み（GitHub＝fork作成と更新に使う・無料）。

5. 今後の検討（提案）：3層構成で dev/prod parity

「私の本番も、ユーザーと同じ GitHub→Cloudflare（Workers Builds） 経路で動かす」案。さらに②検証環境も同じ Workers Builds 経路にして、検証と本番のデプロイ機構を完全に一致させます。これにより「ローカルでは通るのに本番だけ落ちる」という環境差由来のエラーが消え、リスク①②（自動プロビジョニング・マイグレーション・デプロイコマンド）を本番に触れずに②で先に踏んで潰せます。現在のローカルからのリモートデプロイ（npm run deploy --remote 相当）は廃止され、ローカルは論理検証（vitest／sealed wrangler dev）専用になります。

①は残します（降格するのは「ローカルからのCLIリモートデプロイ経路」だけで、ローカルでの開発・テストはむしろ速い反復の主役）。②は staging ブランチ（または検証用fork）に専用の staging Worker を Workers Builds で接続し、テスト用Discordアプリ＋自動生成D1に向けます。ADR 0008 の「検証はデプロイ済みテスト機で」の精神は維持しつつ、手段をCLIから Workers Builds に更新するためADR 0008 の追補（または新ADR）が必要です。なお Workers Builds はローカル .env を読まずシークレットをダッシュボード管理にするため、ADR 0008 が恐れた「本番値混入による誤投稿」リスクはむしろより完全に解消されます。

守るべき規律：「②で検証 → main へ昇格（③へ反映）」「DB変更ありの前にD1バックアップ」「③が追うブランチは安定版に保つ」。複数イベント（Choiemu・他）は1インスタンスのマルチサーバー（guild_id・ADR 0004）で共存でき、イベントごとにデプロイを分ける必要はありません。
